1. Visão geral

O MOVEPLUSAPP (também referido na documentação como Move+) é uma solução cross-platform (iOS / Android) desenhada para instituições de cuidados continuados a idosos. A plataforma cobre todo o ciclo: onboarding via convites, monitorização clínica diária, resposta a incidentes (quedas e SOS), acompanhamento de feridas e coordenação de equipas (cuidadores, clínicos, administração, profissionais externos).

O projeto está organizado como um monorepo com três pacotes interligados que partilham tipos através de moveplus-shared. Existe ainda um pequeno serviço auxiliar all_app_move_bl/ (um health-check Express minimal) e a pasta public/ que serve uploads e avatares por defeito.

2. Stack tecnológica

Frontend (APP)

  • React Native 0.81 + Expo SDK 54 (gestão de plataforma e build)
  • TypeScript 5.9 com path aliases (@src, @navigation, @stores, ...)
  • React Navigation 6 — stacks aninhados por role
  • Zustand — estado global leve (auth, dashboards, listas)
  • Axios — cliente HTTP com interceptors para refresh de token
  • React Native Paper — design system base
  • expo-sensors / expo-task-manager / expo-background-fetch — deteção de quedas em background
  • expo-notifications — push notifications
  • i18next + react-i18next + expo-localization — PT / EN em runtime
  • react-native-gifted-charts, react-native-body-highlighter, react-native-svg — gráficos e diagrama corporal
  • expo-print — exportação de relatórios PDF
  • Zod — validação partilhada com o backend

Backend (SERVER)

  • Node.js + Express 4 em TypeScript
  • Prisma 6 sobre PostgreSQL (~30 modelos, 12 enums)
  • JWT (access) + Refresh Tokens persistidos em BD
  • bcryptjs — hashing de passwords
  • cookie-parser — refresh token em httpOnly cookie
  • Multer — upload de fotos (lesões, avatares, feridas)
  • nodemailer — emails (reset password, convites)
  • expo-server-sdk — envio de push para o serviço Expo (APNs/FCM)
  • Zod — validação de requests

Outros

  • shared/ — entidades, enums, DTOs (request/response) e modelos de notificação
  • Makefile — automação de dev, expo, db, shared, dist
  • Railway / nixpacks — deploy do backend

8. Modelo de dados (Prisma · PostgreSQL)

Total: 699 linhas, com ~30 modelos e 12 enums. Resumo dos blocos:

👤 Identidade

User central (username, email, password, role, avatarUrl, pushToken, password reset). Liga-se a 1-1 com Elderly · Caregiver · Clinician · InstitutionAdmin · Programmer · ExternalPersonnel. Sessões longas via RefreshToken.

🏢 Instituição

Institution agrupa idosos e staff. Invitation permite onboarding por token (PENDING → ACCEPTED / PROFILE_INCOMPLETE / EXPIRED / CANCELLED).

🚨 Quedas e SOS

FallDetection (deteção bruta do sensor) → FallOccurrence (ocorrência tratada com lesões, fotos, falso alarme, medidas tomadas). SosOccurrence manual, com possibilidade de marcar wasActualFall. FallRisk com score temporal.

💊 Saúde

Medication, Pathology, Measurement (11 tipos clínicos com unidade e status colorido), Assessment, Device e Session de dispositivos.

🩹 WoundTracking

Histórico em árvore (parentTrackingId) de feridas associadas a quedas/SOS ou independentes, com fotos e localizações corporais.

📅 Agenda

CalendarEvent com tipos (Appointment, Physiotherapy, NursingCare, Bath, Meal, Activity, Other). StaffWorkSchedule (slots por dia) e StaffTimeOff (férias/folga/baixa). ElderlyAbsence para ausências de residentes.

🔑 Acesso externo

ExternalProfessional (sem conta), ExternalAccessToken (16-hex, 24h) e ExternalVisitNote (observações pós-visita) — permite consultas sem registo.

🔔 Notificações

Notification persiste em BD (type, title, body, JSON data, read) com índices por user e read. TimelineActivity regista o audit-trail por instituição.

9. Perfis de utilizador

O UserRole determina o navigator raiz (em AppNavigator.tsx) e as permissões no backend.

ELDERLY 📱 paciente

  • Auto-monitorização de saúde
  • SOS manual
  • Deteção automática de quedas em background
  • Calendário pessoal

CAREGIVER cuidador

  • Gestão diária dos residentes
  • Adicionar medições/medicamentos/patologias
  • Tratar quedas e SOS
  • Horário próprio + pedidos de folga

INSTITUTION_ADMIN admin

  • Visibilidade total da instituição
  • Convites de novos membros
  • Aprovação/rejeição de folgas
  • Calendário e agenda da equipa

CLINICIAN clínico

  • Pesquisa de idosos
  • Pedidos de acesso a dados
  • Avaliações clínicas

PROGRAMMER sistema

  • Manutenção / debug
  • Acesso a homepage interna

External (sem conta) token

  • Não é um UserRole
  • Acede via token expirável a perfis específicos
  • Submete notas pós-visita

11. Fluxos chave end-to-end

11.1 Login (com refresh)

  1. UI LoginScreen.tsx chama useAuthStore.login()
  2. store authStore.ts → authService.login()
  3. http ApiService.ts faz POST /api/auth/login (refresh token vem em httpOnly cookie)
  4. api authController.ts valida credenciais → emite JWT + refresh + grava em RefreshToken
  5. store guarda user e config; persiste no AsyncStorage
  6. side Se role = ELDERLY → startFallDetection()
  7. push NotificationProvider regista token Expo → POST /api/notifications/register-token
  8. nav AppNavigator deteta novo role e troca o root navigator

11.2 Queda detetada automaticamente

  1. 📱 idoso Acelerómetro lê magnitude > 2.5g (accelerometerService.ts)
  2. Aguarda 3s de imobilidade → confirma deteção
  3. Abre IncidentConfirmationModal com countdown 30s (incidentConfirmationStore.ts)
  4. "Preciso de Ajuda" (ou timeout) → POST /api/fall-occurrences
  5. api fallOccurrenceController.ts cria registo Prisma
  6. Cria TimelineActivity via timelineService.ts
  7. Procura admins/cuidadores da instituição → notifyFallOccurrence()
  8. notificationService.ts envia push via expo-server-sdk (chunks ≤100)
  9. 📱 admin Recebe notificação. Toque → NotificationProvider faz navigationRef.dispatch para FallOccurrenceScreen
  10. Admin trata, faz upload de foto de lesão (uploadMiddleware.ts), exporta PDF

11.3 Acesso externo (sem conta)

  1. Admin cria CalendarEvent com externalProfessionalId
  2. Chama POST /api/external-access/generate (autenticado)
  3. externalAccessController.ts gera token hex 16 chars (24h validade), grava em ExternalAccessToken
  4. Token enviado por email/SMS (fora da app)
  5. Profissional externo acede via /api/external-access sem autenticação (rota montada antes do /api protegido em server.ts)
  6. Vê ficha do idoso → submete ExternalVisitNote
